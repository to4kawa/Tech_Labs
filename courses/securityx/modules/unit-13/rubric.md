# Unit 13 採点表（固定）

## 採点方針
- 合計100点、**80点以上かつ必須項目を全達成**で合格。
- 必須項目を1つでも満たさない場合は総合点に関係なく不合格。

## 必須項目（Gate）
1. ログ根拠のない主張を提出物に含めない。
2. 「端末表示/端末内ログのみ」で判定していない。
3. 迂回経路が残る状態で「使える」判定を出していない。
4. 例外運用に期限・承認・棚卸しがある。

## 配点
| 評価観点 | 配点 | 判定基準 |
|---|---:|---|
| 1. 全通信棚卸し | 20 | L3/L4・DNS・HTTP/HTTPS・その他を証跡付きで網羅 |
| 2. allowlist品質 | 15 | proto/port/dst/理由が揃い、更新履歴が追跡可能 |
| 3. 制御段階の証明 | 20 | 規定6段の各段で「止まる/残る」を差分提示 |
| 4. MITM比較（TLS+非TLS） | 15 | A/B/Cの差分と成立条件を説明可能 |
| 5. 迂回検知と封鎖 | 15 | 抜け道検知→対策追加→再検証を実証 |
| 6. 運用設計（例外/監査/継続） | 15 | 例外ルール、承認証跡、棚卸し運用が実装済み |

## 減点ルール
- 証跡リンク切れ: 1件につき -3点
- 時系列不整合の放置: -5点
- 変更理由未記載: 1件につき -2点
- 再検証なしで完了判定: -10点

## 提出物チェック（提出時）
- [ ] 01_inventory: 全通信棚卸し（ログ根拠付き）
- [ ] 02_allowlist: 許可通信定義（理由列あり）
- [ ] 03_bypass_map: 迂回経路と対策マップ
- [ ] 04_mitm_tls_non_tls: TLS/非TLS比較
- [ ] 05_judgement_criteria: 使える/使えない判定基準
- [ ] 06_exception_ops: 例外運用ルールと実例

## 合否コメントテンプレ
- 合格例: 「外側ログで通信成立条件を説明でき、迂回封鎖と例外統制が両立している。」
- 不合格例: 「根拠ログなしの主張があり、迂回残存にもかかわらず許可判定を出している。」
